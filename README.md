# Blood Donation Platform — REST API

A backend REST API for a community-driven blood donation platform. Built with Laravel, PHP 8, PostgreSQL 17, and JWT authentication. Fully containerized with Docker.

## Getting Started

Requires Docker and Docker Compose only. No local PHP or PostgreSQL installation needed.

```bash
git clone https://github.com/your-org/blood-donation-api.git
cd blood-donation-api
```

Download the .env file from here and paste it inside src folder
https://drive.google.com/file/d/1lVcqXjumny18BngJyIXrols8H5X91DzE/view?usp=sharing

Make sure the file is named as **.env**

```bash
docker compose up -d --build

docker exec -it blood_app bash
composer install
php artisan key:generate
php artisan jwt:secret

php artisan migrate:fresh --seed

exit
```

Verify the setup: `GET http://localhost:8000/api/health` should return a success response.

**Default admin account (seeded automatically):**

```
Email:    admin@blood.com
Password: password123
```

## Database Schema

```
users                       core account (role: admin / user / organization)
user_profiles               donor profile (blood group, location, availability, trust score)
organizations               NGO profile (verification status, contact)
organization_documents      uploaded docs for org verification
donation_requests           blood requests posted by users
donation_request_recipients donors selected per request, their response, dual confirmation state
events                      blood collection events by organizations
event_registrations         donor registrations per event with attendance tracking
payments                    payment records (dummy, for audit — gateway not integrated)
reports                     fraud reports on users, donation requests, and events
```

All tables except `payments` include soft delete support and full audit columns: `deleted_at`, `created_by`, `updated_by`, `deleted_by`.

The `reports` table uses explicit nullable foreign keys (`target_user_id`, `target_donation_request_id`, `target_event_id`) instead of a polymorphic pattern, with `ON DELETE SET NULL` so report history is preserved when targets are deleted.

## Authentication

All protected routes require a JWT token in the `Authorization` header:

```
Authorization: Bearer <token>
```

Tokens are issued on registration and login. On logout the token is blacklisted server-side and cannot be reused.

**Register:**

```json
POST /api/auth/register
{
  "name": "John Doe",
  "email": "john@example.com",
  "password": "password123",
  "role": "user"
}
```

**Login:**

```json
POST /api/auth/login
{
  "email": "john@example.com",
  "password": "password123"
}
```

## Role System

| Role           | Description                                          |
| -------------- | ---------------------------------------------------- |
| `user`         | Can donate blood, post requests, register for events |
| `organization` | NGO that creates blood collection events             |
| `admin`        | Full platform control                                |

Roles are assigned at registration. Admin accounts cannot be created via the API.

Organization accounts start as `pending`. Pending organizations can log in but are restricted to their profile page until an admin approves them. Rejected organizations are blocked at login.

## Trust Score System

Every user profile has a `trust_score` between `0.00` and `1.00`. The system controls this value, it cannot be edited through the API.

| Action                                          | Change |
| ----------------------------------------------- | ------ |
| Both donor and requester confirm a donation     | +0.05  |
| Donor registered for an event but marked absent | -0.10  |

The score is capped at `1.00` and floored at `0.00`. Donor search results are sorted by trust score descending.

**Dual confirmation requirement:** Trust score only increases when both the donor confirms via `confirm-donated` and the requester confirms via `confirm-received`. A single-party confirmation has no effect on the score. The `response_status` on the recipient record only moves to `donated` once both sides confirm.

## Audit Fields

All tables except `payments` use the `HasAuditFields` trait, which wraps Laravel's `SoftDeletes` and auto-populates the following on every write:

| Column       | Populated on                   |
| ------------ | ------------------------------ |
| `created_by` | Record creation                |
| `updated_by` | Every update                   |
| `deleted_by` | Soft delete                    |
| `deleted_at` | Soft delete (standard Laravel) |

Values are sourced from `auth('api')->id()` and are `null` for unauthenticated operations such as seeders.

## Running Tests

Tests run against a separate database so they never affect development data.

**One-time test database setup:**

```bash
docker exec -it blood_postgres psql -U blood_user -d blood_donation -c "CREATE DATABASE blood_donation_test;"
```

**Run all tests:**

```bash
docker exec -it blood_app php artisan test
```

**Run a specific file:**

```bash
docker exec -it blood_app php artisan test --filter AuthTest
```

Current coverage: 95 tests, 188 assertions across 10 test files covering every section of the API.

## Daily Workflow

```bash
# Start containers

# Run tests
```
